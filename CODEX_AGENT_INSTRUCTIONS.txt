Build Task 1 Performance UI for Customer and Product.

Project context:
This is for a Cerebras-style performance dashboard challenge. The app must turn internal .xlsx performance projection sheets into something two audiences can act on:
1. A customer or PM who needs a simple go / caution / no-go signal.
2. An internal engineer who needs to sanity-check model performance projections.

Use the existing React + TypeScript project structure.

Use:
- Vite
- React
- TypeScript
- SheetJS/xlsx for Excel parsing
- Recharts for charts
- No backend
- Everything should run in the browser

Core requirement:
A reviewer must be able to upload one or more .xlsx perf sweep files live in the deployed app, including unknown future models like Model L, and the app must render and compare the views without code changes.

Important:
Do not hardcode Model A, Model B, Model C, etc.
Do not hardcode only seven profiles.
Detect model name and profile number dynamically from filename and/or sheet data.
Support uploads like:
- Model A profile 1.xlsx
- Model B profile 1.xlsx
- Model L profile 7.xlsx

Excel parsing:
Parse uploaded .xlsx files.
Prefer a sheet named Summary if it exists.
If Summary does not exist, use the first sheet with usable rows.
Extract and normalize these possible columns:
- Input Length
- Output Length
- Cache %
- Batch Size
- Throughput (t/s)
- Uncached Throughput (t/s)
- Cached Throughput (t/s)
- Prompt-only Throughput
- Gen-only Throughput
- TTFT (ms)
- TTIF / TTIT if present
- RPM
- Target Max if present
- Real Prompt if present
- Any millisecond or latency-related columns if present

Normalize all rows into a shared TypeScript schema:
- id
- fileName
- modelName
- profileName
- profileNumber
- workloadLabel
- inputLength
- outputLength
- cachePercent
- batchSize
- throughput
- uncachedThroughput
- cachedThroughput
- promptOnlyThroughput
- genOnlyThroughput
- ttftMs
- ttifMs
- rpm
- targetMax
- realPrompt
- raw

Add column alias handling because files may use slightly different column names.

Profile meaning:
Treat profiles as workload scenarios, not different models.
Create friendly workload labels based on input length, output length, and cache percent:
- short input + short output + high cache = Interactive Chat
- long input + long output = Long Document Summarization
- medium input + medium output = Assistant / RAG
- short input + long output = Long Generation
- very long input + short output = Extreme Long Context Retrieval
- long input + medium/long output = Agentic / Heavy Mixed Workload
Use these labels as helpful UI hints, not hard truth.

Developer UI:
Create a developer dashboard with six layers.

Layer 1: Quick Health Overview
- Show model health score
- Show status badges: Excellent, Stable, Moderate Risk, Unstable
- Show key metrics: throughput, TTFT, RPM, stability score
- Show workload labels
- Show risk indicators for problematic profiles
- Goal: engineer understands model condition within 5 seconds

Layer 2: Workload Behavior
- Show what each profile/workload is good for
- Use workload cards instead of only profile numbers
- Show Best For labels such as Chat, Long Context, Agentic, Summarization
- Show whether workload is prompt-heavy, generation-heavy, cache-heavy, or latency-sensitive
- Goal: engineer understands what type of application this model supports

Layer 3: Scaling and Performance Visualization
- Charts:
  - Throughput vs Batch Size
  - TTFT vs Batch Size
  - RPM vs Batch Size
  - Cached vs Uncached Throughput
  - Prompt-only vs Gen-only Throughput if available
- Highlight scaling behavior, saturation, and latency spikes
- Goal: engineer sees whether the model scales or collapses under load

Layer 4: Outlier and Anomaly Detection
- Automatically flag:
  - throughput drops
  - throughput flattening
  - TTFT spikes
  - RPM instability
  - non-numeric values
  - missing columns
  - duplicate batch rows
  - weird cache behavior
  - saturation at high batch sizes
- Show anomaly cards and highlight problematic profiles
- Goal: engineer instantly knows where to investigate

Layer 5: Deep Diagnostics
- Show raw normalized rows in an expandable section
- Show file parsing summary
- Show detected model/profile metadata
- Show validation warnings
- Show metric-level details for each batch
- Goal: engineer can debug and sanity-check the projection

Layer 6: Cross-Model Same-Profile Comparison
- This is very important.
- Compare the same profile across different models:
  - Model A Profile 1 vs Model B Profile 1 vs Model C Profile 1
- Do not compare Model A Profile 1 against Model A Profile 7 as if one is better, because profiles represent different workloads.
- Add a profile selector.
- For selected profile, compare all uploaded models side by side.
- Show winner badges for:
  - best throughput
  - best TTFT
  - best RPM
  - best stability
  - best overall
- Add a “Why this model wins” explanation panel.
- Goal: answer which model handles this workload best.

Customer / PM UI:
Create a separate customer-facing view that avoids technical complexity.

Customer Layer 1: Simple Recommendation
- Show recommended model
- Show GO / CAUTION / NO-GO status
- Show confidence level
- Show simple reason
- Example: “Model A is recommended for Interactive Chat because it has low first-token latency and strong request capacity.”

Customer Layer 2: Workload Understanding
- Avoid only saying Profile 1, Profile 2.
- Use human labels:
  - Interactive Chat
  - Coding Assistant
  - Long Document Analysis
  - Summarization
  - High Traffic Serving
  - Agentic Workflow
- Explain what the workload means in one short sentence.

Customer Layer 3: Visual Evidence
- Show clean charts with minimal clutter.
- Highlight the best model visually.
- Use simple captions:
  - “Lower is better”
  - “Higher is better”
  - “This model stays stable as load increases”
- Avoid raw spreadsheet-style tables in the customer view.

Customer Layer 4: Why This Recommendation?
- Add a clickable “Why?” button for each recommendation.
- Open an evidence panel/modal/drawer.
- Show:
  - key strengths
  - key risks
  - supporting charts
  - simple explanation
- The user should understand why Model A is recommended and why Model D may be risky.

Customer Layer 5: Production Readiness
- Show:
  - Stable / Risky badge
  - latency quality
  - traffic readiness
  - scalability confidence
  - warnings for risky workloads
- Goal: customer or PM can decide if the model is safe enough for their use case.

Scoring:
Create a scoring system from 0 to 100.
Use:
- higher throughput is better
- higher RPM is better
- higher gen-only throughput is better
- lower TTFT is better
- stable scaling gets a bonus
- cache efficiency gets a bonus when cache percent is high
- missing metrics reduce confidence
- anomaly count reduces score

Status:
- score >= 75: GO
- score >= 50: CAUTION
- below 50: NO-GO

Stability logic:
- If throughput drops as batch size increases, flag it.
- If TTFT jumps sharply, flag it.
- If RPM drops or plateaus unexpectedly, flag it.
- If cached throughput is not meaningfully better when cache percent is high, flag it.
- If batch scaling flattens, mark saturation.

UI design:
- Create two main tabs:
  - Customer / PM View
  - Engineer View
- Use a clean modern dashboard style.
- Make the UI visual-first.
- Avoid huge tables as the main interface.
- Use cards, badges, charts, small explanations, and expandable details.
- Make it easy to understand at a glance.
- Use responsive layout.
